# Target Folder Structure

The canonical layout every Node.js + TypeScript + Express + Mongoose project
generated by this skill should converge toward. Names are lowercase kebab where
possible, singular for "this file describes one thing" (service, repository),
plural where the file is a namespace (routes).

```
server/src/
├── composition/
│   └── container.ts                # DI container — ONLY place `new` appears
│
├── controllers/                    # HTTP orchestration
│   ├── trade.controller.ts
│   ├── rule.controller.ts
│   └── analytics.controller.ts
│
├── errors/                         # Typed error hierarchy
│   ├── AppError.ts                 # Base
│   ├── NotFoundError.ts
│   ├── ValidationError.ts
│   ├── UnauthorizedError.ts
│   ├── ForbiddenError.ts
│   └── ConflictError.ts
│
├── interfaces/                     # Contracts — no implementations here
│   ├── trade.interfaces.ts
│   ├── rule.interfaces.ts
│   ├── storage.interfaces.ts
│   └── base.repository.ts          # Generic CRUD contract
│
├── mappers/                        # Pure DTO transforms
│   ├── trade.mapper.ts
│   └── rule.mapper.ts
│
├── middleware/                     # Cross-cutting concerns
│   ├── authMiddleware.ts
│   ├── asyncHandler.ts
│   └── errorHandler.ts             # Mount LAST
│
├── models/                         # Mongoose schemas + I<X> interfaces
│   ├── Trade.ts
│   └── Rule.ts
│
├── repositories/                   # Data access ONLY
│   ├── trade.repository.ts
│   └── rule.repository.ts
│
├── routes/                         # HTTP → controller binding
│   ├── trades.ts
│   └── rules.ts
│
├── services/                       # Business logic
│   ├── trade.service.ts
│   ├── rule.service.ts
│   ├── analytics.service.ts
│   └── storage/                    # Sub-folder for related strategies
│       ├── storage.factory.ts
│       ├── googleDrive.storage.ts
│       └── local.storage.ts
│
├── types/
│   └── index.ts                    # Shared DTOs, ApiResponse<T>, pagination
│
├── validators/                     # Zod schemas + middleware
│   ├── trade.validator.ts
│   └── common.validator.ts
│
├── db.ts                           # Mongo connection bootstrap
└── index.ts                        # App entry — mount middleware + routes only
```

## Why this shape

**One file, one concern** — you can always `grep trade.service` and know exactly
which file business logic lives in. No hunting through `utils/` or `helpers/`.

**Layer names are nouns** — `controllers/`, `services/`, `repositories/`. Each
matches a SOLID "S" responsibility.

**Interfaces + implementations in separate folders** — forces you to think of
them as two different things. Imports read cleanly: "this file depends on the
*contract* of trade, not the class".

**`composition/container.ts` is the one concession to centralization** — every
other file avoids concrete dependencies. This is DIP done pragmatically, without
a heavyweight IoC library.

**`storage/` as a sub-folder of `services/`** — related strategies live
together. When you add S3, it joins the neighborhood.

## Naming conventions

| File | Pattern | Example |
|------|---------|---------|
| Interface | `<module>.interfaces.ts` | `trade.interfaces.ts` |
| Model | `<Module>.ts` (PascalCase) | `Trade.ts` |
| Repository | `<module>.repository.ts` | `trade.repository.ts` |
| Mapper | `<module>.mapper.ts` | `trade.mapper.ts` |
| Service | `<module>.service.ts` | `trade.service.ts` |
| Validator | `<module>.validator.ts` | `trade.validator.ts` |
| Controller | `<module>.controller.ts` | `trade.controller.ts` |
| Route | `<modules>.ts` (plural) | `trades.ts` |
| Error | `<Name>Error.ts` (PascalCase) | `NotFoundError.ts` |

## ESM import extensions

If `tsconfig.json` has `"module": "NodeNext"` or `"ESNext"`, **imports MUST
include the `.js` extension** even when pointing to `.ts` source:

```ts
import { TradeService } from '../services/trade.service.js'; // ✅ correct
import { TradeService } from '../services/trade.service';    // ❌ fails at runtime
```

Always run the `node-verify-import` skill after refactors to catch this.
